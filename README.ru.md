# bounce

**bounce** — быстрый файловый архиватор без внешних зависимостей, написанный на Rust. В основе лежит кодек **Big Bounce**: данные «схлопываются» при сжатии и точно «отскакивают» обратно при распаковке — по аналогии с космологической моделью Большого отскока.

- Чистый Rust, **ноль внешних зависимостей** (только стандартная библиотека).
- Автоматический выбор метода сжатия для каждого файла.
- Многопоточное сжатие и распаковка крупных файлов.
- Контроль целостности каждого файла по **CRC-32**.
- Защита от path traversal при распаковке.
- Расширение архивов: **`.bnc`**.

---

## Сборка

```bash
cd bounce
cargo build --release
# бинарник: target/release/bounce
```

### Нативная сборка под Apple Silicon (ARM64)

Если вы работаете на Mac с процессором Apple Silicon, по умолчанию ваш тулчейн Rustup может быть настроен на эмуляцию x86_64 (Rosetta 2). Для получения максимальной производительности соберите архиватор нативно под архитектуру `aarch64-apple-darwin`:

```bash
cargo build --release --target aarch64-apple-darwin
# бинарник: target/aarch64-apple-darwin/release/bounce
```

Установка в систему (опционально):

```bash
cargo install --path .
```

## Команды

```bash
bounce <команда> [опции] <архив> [файлы...]
```

| Команда | Алиас | Назначение |
|---------|-------|------------|
| `create`  | `c` | Создать архив из файлов и каталогов |
| `extract` | `x` | Распаковать архив (полностью или выбранные файлы) |
| `list`    | `l` | Показать содержимое архива |
| `test`    | `t` | Проверить целостность архива (CRC-32) |
| `help`    | `h` | Справка |
| `version` | `v` | Версия |

### Опции

| Опция | Описание |
|-------|----------|
| `-o, --output <dir>` | Каталог для распаковки (по умолчанию — текущий) |
| `-c, --stdout` | Выводить распакованные файлы прямо в stdout |
| `-f, --force` | Перезаписывать существующие файлы при распаковке |
| `-v, --verbose` | Показывать прогресс по каждому файлу |
| `-q, --quiet` | Не выводить итоговую строку |

## Примеры

Создать архив из файла и каталога:

```bash
bounce c backup.bnc report.pdf photos/
```

Посмотреть содержимое:

```bash
bounce l backup.bnc
```

```
    Original        Stored   Ratio      Method  Name
----------------------------------------------------------------
      54.0 KB       16.3 KB   30.2%       plain  report.pdf
       1.2 MB      420.0 KB   34.1%    shuf+blk  photos/img01.png
----------------------------------------------------------------
       1.3 MB      436.3 KB   33.8%              2 file(s)
```

Проверить целостность:

```bash
bounce t backup.bnc
# backup.bnc: OK (2 file(s) verified)
```

Распаковать всё в каталог `restored/`:

```bash
bounce x backup.bnc -o restored/
```

Распаковать один файл:

```bash
bounce x backup.bnc report.pdf -o restored/
```

## Теория и научные основы: модель Большого отскока и Золотое сечение

В основе кодека Big Bounce лежат фундаментальные аналогии с теоретической физикой и математическим анализом разреженных структур.

### 1. Космологическая модель Большого отскока (Big Bounce)
В отличие от классической теории Большого взрыва, предполагающей сингулярность (состояние с бесконечной плотностью и нулевым объёмом, где любая информация о предыдущих состояниях материи безвозвратно теряется), петлевая квантовая космология предлагает модель **Большого отскока** (Big Bounce). В ней Вселенная перед фазой расширения проходит через фазу максимального сжатия до планковских плотностей, где квантово-гравитационные силы отталкивания предотвращают коллапс в сингулярность и заставляют Вселенную «отскочить» обратно, полностью **сохраняя всю информацию**.

Перенесённый на теорию информации, этот принцип формулируется так:
> Любое представление данных можно обратимо сжать к состоянию минимальной избыточности («точка отскока») и затем точно развернуть обратно, при условии, что сжатие порождается строго **биективными** преобразованиями, не теряющими ни одного бита информации.

### 2. Порог золотого сечения ($\rho_c \approx 0.382$)
Для оптимизации и выбора стратегий сжатия разреженных информационных потоков (например, весов глубоких нейросетей после квантования и прунинга) архиватор использует критический порог плотности $\rho_c$, основанный на **золотом сечении**:
$$\rho_c = \frac{3 - \sqrt{5}}{2} \approx 0.382$$
Этот порог является сопряжённым значением золотого сечения и выводится из квадратного уравнения:
$$q^2 + q - 1 = 0$$

В теории сжатия разреженных потоков данных (концепция VMF/NVG) величина $\rho_c \approx 0.382$ — это точная математическая граница энтропии, при которой дальнейшая дедупликация через стандартный статический кодбук перестает давать выигрыш по сравнению со специализированным энтропийным кодером.
- Если локальная плотность данных $\rho < \rho_c$, архиватор динамически адаптирует структуру кодбука (dual-codebook) и применяет EMA-сглаживание плотности (EMA density) с адаптивным гейтингом (adaptive gain gate).
- Использование порога золотого сечения позволяет кодеку `bounce` мгновенно определять границы сжимаемости блоков данных и отсекать избыточные вычислительные проходы, что **ускоряет сжатие до 2.3×** за счёт своевременного пропуска неподходящих методов.

## Как работает сжатие

Кодек Big Bounce — самодостаточная реализация в стиле DEFLATE, развивающая изложенные выше принципы:

1. **LZ77** со скользящим окном 64 КБ находит повторяющиеся последовательности.
2. **Хаффман** кодирует символы и коды расстояний (своё оптимальное дерево на каждый блок 32 КБ).
3. **Byte-shuffle** трансформации (stride = 2 и 4) раскрывают структурную избыточность бинарных данных (например, весов нейросетей в формате float32).

Для каждого файла перебираются методы `plain`, `blocked`, `shuf+defl`, `shuf+blk`, `shuf2+defl`, `shuf2+blk` и выбирается наилучший по коэффициенту сжатия. Если файл несжимаемый (например, уже сжатые или случайные данные), он сохраняется без изменений (метод `stored`), поэтому архив никогда не больше суммы исходных файлов плюс небольшие заголовки.

Большие файлы (> 1 МБ) обрабатываются только блочными методами, что позволяет распараллеливать работу по числу ядер CPU.

> **Ограничение окна.** LZ-окно равно 64 КБ, поэтому повторы, разнесённые на большее расстояние, не дедуплицируются. На данных с дальними повторами (очень крупные шаблоны, дампы БД) `bounce` уступает компрессорам с большим словарём (`xz`, `zstd --long`, `brotli`), но выигрывает в скорости и простоте.

Подробное описание формата контейнера — в [FORMAT.md](FORMAT.md).

**Честные выводы:**

- **Распаковка:** Нативная распаковка `bounce` под архитектуру ARM64 достигает **~1.31 ГБ/с** на крупных файлах. Несмотря на огромный прирост скорости по сравнению с эмуляцией, она всё же уступает нативным стриминговым компрессорам `lz4` и `brotli` (которые достигают 3–4 ГБ/с) из-за накладных расходов на поблочное декодирование Хаффмана и полный пересчет CRC-32.
- **Сжатие:** На крупных бинарных данных `bounce` быстр за счёт многопоточности и маршрутизации методов, но LZ77-окно в 64 КБ ограничивает дедупликацию дальних повторов. По этой причине на текстовом файле объёмом 500 МБ архиваторы с большими словарями (`zstd`, `xz`, `brotli`) сжимают данные практически в ноль, тогда как `bounce` удерживает коэффициент сжатия на уровне 33.8%.
- **Безопасность:** На несжимаемых данных `bounce` гарантированно исключает раздувание архива за счёт переключения в режим хранения сырых байт (`stored`).
