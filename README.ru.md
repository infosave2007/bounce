# bounce

![Build Status](https://img.shields.io/badge/build-passing-brightgreen)
![License](https://img.shields.io/badge/license-Apache%202.0-blue)
![Rust](https://img.shields.io/badge/rust-1.70%2B-orange)

**bounce** — быстрый многопоточный файловый архиватор на чистом Rust без внешних зависимостей. В основе лежит кодек **Big Bounce** с умной динамической маршрутизацией методов сжатия и оптимизированным байт-шаффлингом для крупных бинарных данных. Разработан для обеспечения максимальной скорости работы и безопасности.

## Основные возможности

- **Ноль внешних зависимостей**: Только стандартная библиотека Rust.
- **Кроссплатформенность**: Полная поддержка macOS, Linux и Windows (на архитектурах x86_64, aarch64, x86).
- **Асинхронный конвейер (Pipeline)**: Параллельная работа диска и процессора. Пока с диска читается следующий блок данных, CPU-токи уже распаковывают предыдущий.
- **Smart Routing (Умная маршрутизация)**: Анализирует энтропию файла "на лету" и автоматически выбирает оптимальную стратегию сжатия (LZ77, алгоритм Хаффмана, Byte-Shuffling или прямое копирование).
- **Безопасность**: Защита от атак типа path traversal. Контроль целостности файлов по контрольной сумме CRC-32.
- **Pre-flight проверки диска**: Архиватор заранее проверяет наличие свободного места перед распаковкой или упаковкой, предотвращая сбои из-за нехватки памяти.
- Расширение архивов: **`.bnc`**.

## Основные сценарии использования (Ниши)

`bounce` специально спроектирован для работы с массивными и плотными бинарными файлами, где традиционные архиваторы упираются в узкие горлышки скорости или потребления ОЗУ.

- **Большие языковые модели (LLM) и нейросети**: Идеально подходит для сжатия гигабайтных весов искусственного интеллекта (`.safetensors`, `.gguf`, `.pt`, `.bin`). Алгоритм byte-shuffle математически "выравнивает" `float16`/`float32` структуры, обеспечивая экстремальную скорость распаковки (~1.3 ГБ/с), что критично для пайплайнов быстрой загрузки моделей в память видеокарты.
- **Big Data и научные вычисления (HPC)**: Быстрое архивирование бинарных датасетов, дампов памяти, состояний симуляций и телеметрии, где максимальная пропускная способность I/O (ввода-вывода) диска важнее всего.
- **GameDev и упаковка ассетов**: Мгновенная упаковка и распаковка тяжелых игровых ресурсов (текстур, геометрии, аудио-банков) благодаря асинхронному пулу буферов и полному отсутствию тяжеловесных внешних зависимостей.

## Установка

### Сборка из исходников

```bash
git clone https://github.com/your-repo/bounce.git
cd bounce
cargo build --release
```
Бинарный файл появится по пути `target/release/bounce`.

### macOS Apple Silicon (M1/M2/M3/M4)
Для максимальной производительности на Mac с процессорами ARM64 убедитесь, что вы компилируете нативно, а не через эмулятор Rosetta 2:

```bash
cargo build --release --target aarch64-apple-darwin
```

### Установка в систему
```bash
cargo install --path .
```

## Использование

```bash
bounce <команда> [опции] <архив> [файлы...]
```

### Команды

| Команда | Алиас | Назначение |
|---------|-------|------------|
| `create`  | `c` | Создать архив из файлов и каталогов |
| `extract` | `x` | Распаковать архив (полностью или выбранные файлы) |
| `list`    | `l` | Показать содержимое архива |
| `test`    | `t` | Проверить целостность архива (CRC-32) |

### Опции

| Опция | Описание |
|-------|----------|
| `-1 ... -9` | Уровень сжатия (по умолчанию `-1`). `-1` — максимальная скорость, `-9` — максимальное сжатие за счет большего окна поиска. |
| `-o, --output <dir>` | Каталог для распаковки (по умолчанию — текущий) |
| `-c, --stdout` | Выводить распакованные файлы напрямую в stdout |
| `-f, --force` | Перезаписывать существующие файлы при распаковке |
| `-v, --verbose` | Показывать детальный прогресс по каждому файлу |
| `-q, --quiet` | Отключить вывод итоговой сводки |

### Пример уровней сжатия

*Влияние уровня сжатия на видеофайл размером 50.7 МБ (`.mp4`):*

| Уровень | Окно / Блок | Сжатый размер | Ratio |

| `-2` | 128 КБ / 128 КБ | 49.8 MB | 98.2% |
| `-3` | 256 КБ / 256 КБ | 49.8 MB | 98.2% |
| `-5` | 1 МБ / 1 МБ | 49.0 MB | 96.5% |
| `-6` | 2 МБ / 2 МБ | 47.9 MB | 94.4% |
| `-7` | 4 МБ / 4 МБ | 47.7 MB | 94.0% |
| `-8` | 8 МБ / 8 МБ | 46.8 MB | 92.3% |
| `-9` | 16 МБ / 16 МБ | 46.8 MB | 92.1% |

### Примеры

**Упаковать файлы и папки:**
```bash
bounce c backup.bnc report.pdf photos/
```

**Распаковать архив в конкретную папку:**
```bash
bounce x backup.bnc -o restored/
```

**Посмотреть содержимое архива:**
```bash
bounce l backup.bnc
```

**Проверить целостность архива:**
```bash
bounce t backup.bnc
```

## Архитектура и алгоритмы

`bounce` использует DEFLATE-подобный алгоритм, усиленный механизмами динамической реструктуризации и агрессивной многопоточности:

1. **Smart Routing (Предварительный анализ)**
   Перед сжатием файла `bounce` вычисляет его информационную энтропию Шеннона, битовую плотность и частоту повторов. На базе этого профиля выбирается идеальный вычислительный конвейер (включение/отключение LZ77, включение Byte-Shuffle). Это экономит ресурсы процессора на несжимаемых данных.

2. **Byte-Shuffling для структурных данных**
   Для бинарных массивов (например, веса нейросетей в формате `float32` или `float16`), кодек применяет транспонирование байтов (stride = 2 или 4). Это выравнивает мантиссы и экспоненты чисел друг с другом, обнажая колоссальную структурную избыточность для энтропийного кодирования.

3. **Блочная многопоточность**
   Крупные файлы автоматически разбиваются на независимые блоки (chunks), что позволяет утилизировать все ядра процессора при сжатии и распаковке.

4. **Асинхронный конвейер (Pipeline)**
   Ядро распаковки работает в режиме конвейера с пулом буферов (Buffer Pool): фоновый поток последовательно читает данные с диска, декодирует их параллельно, и по внутренним zero-copy каналам передает главному потоку для проверки CRC и мгновенной записи. Жесткий диск и процессор работают одновременно.

> **Ограничение словаря:** Ради максимальной скорости и минимального потребления оперативной памяти, размер окна поиска LZ77 ограничен 64 КБ. `bounce` феноменально сжимает плотные бинарные файлы и локальные повторы, но на текстовых базах данных может уступать алгоритмам с большими словарями (`zstd`, `xz`).

| zstd -19 | 359.4 MB | 76.2% | 12.4 MB/s | 383.5 MB/s |
